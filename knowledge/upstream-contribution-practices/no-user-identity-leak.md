# No User Identity Leak

## Rule

Files generated by a skill must NOT contain the contributor's local user
directory, username, home path, or any other personally identifying
reference. This applies to any repo the files are committed to — not just
upstream. Private repos can be shared with collaborators, made public
later, or expose machine identity in ways the user didn't intend. If a
personal reference is unavoidable (e.g., the "Implementation" link in an
orientation issue must point to the fork), confirm with the user before
including it.

## Failure Modes

### 1. Absolute local paths in generated content

An AI agent working from `$HOME/p/gh/<owner>/<repo>` may emit paths
like `$HOME/p/gh/<owner>/<repo>/flake.nix` into a README, a comment,
or a CI workflow. Once committed, this path leaks the contributor's
OS username and directory structure.

### 2. Username in install examples or docs

Generated install instructions might reference `github:your-user/repo`
instead of `github:upstream-owner/repo`. This is partially covered by
[Upstream Identity](upstream-identity.md), but the broader case
includes usernames in paths, comments, test fixtures, or error
messages embedded in generated files.

### 3. Fork-specific configuration

A generated workflow or config file might reference the fork's branch
name, the contributor's GitHub token scope, or local environment
variables that only exist on the contributor's machine.

## Prevention

1. **Never hardcode absolute paths** in generated files. Use relative
   paths or upstream-repo-relative references.
2. **Run `scan-artifacts.sh` before committing** — the deterministic
   scan script resolves this machine's actual identity values from the
   environment and system commands (`$HOME`, `whoami`, `hostname`, WiFi
   SSID, DNS search domain) and scans staged files (or specified files)
   for those specific strings. All values are resolved dynamically —
   nothing is hardcoded. It catches the class of bug that hit the Archon
   PR (`.devbox/` with machine-local paths committed, caught only by
   review feedback). Usage:
   ```bash
   # Scan all staged files
   scripts/scan-artifacts.sh
   # Scan specific files the skill generated
   scripts/scan-artifacts.sh flake.nix README.md .github/workflows/nix.yml
   # Show matching lines
   scripts/scan-artifacts.sh --verbose
   ```
   Exit 0 = clean. Exit 1 = findings — review each: fix HARD leaks
   (resolved paths, usernames, WiFi SSID), confirm REVIEW items
   (hostname, DNS domain, literal `$HOME`/`$USER` in non-shell files)
   are legitimate. Re-run until clean. If the user confirms the files
   are for private use only, pass `--private` — HARD findings become
   informational (non-blocking, exit 0) so the commit isn't blocked.
3. **Use `$UPSTREAM_OWNER`/`$UPSTREAM_REPO` variables** for all
   repo references, substituted by text replacement (see
   [gh --body-file](gh-body-file.md)).
4. **Confirm before including any personal reference** — if the
   orientation issue needs a link to the fork branch, show the user
   the exact line and get explicit confirmation before posting.

## What Counts as "Personal"

| Reference | Personal? | Action |
|-----------|-----------|--------|
| `$HOME/...` (resolved path) | Yes | Remove, use relative path |
| `github.com/your-user/repo` in install docs | Yes | Replace with upstream |
| `github.com/your-user/repo/tree/feat-branch` in issue | Yes, but necessary | Confirm with user |
| `$HOME`, `~/.config/...` in user-facing docs | Context-dependent | Use generic `~` or confirm |
| Local env var names (`MY_TOKEN`) | Yes | Remove |
| OS username in file paths or comments | Yes | Remove |
| WiFi SSID (network name) | Yes | Remove — reveals physical location |
| DNS search domain (e.g. tailnet name) | Context-dependent | Remove unless file is network config |
| Hostname / machine name | Context-dependent | Remove unless file is network config |
