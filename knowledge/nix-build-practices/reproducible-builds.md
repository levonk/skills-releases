# Reproducible Builds

## Failure Mode

Without lock files, Nix builds use whatever nixpkgs version is current, causing
different outputs across machines and time. Unpinned dependencies lead to "it
built yesterday but not today" failures.

## Practice

**Commit lock files for reproducibility.**

### flake.lock

- Generated by `nix flake lock`
- Pins exact nixpkgs commit hash for each input
- Must be committed to git
- Updated deliberately with `nix flake update`

### devbox.lock

- Generated by `devbox install` or `devbox add`
- Pins exact package versions
- Must be committed to git
- Updated with `devbox update`

### Deterministic Builds

With lock files committed:
- Same nixpkgs commit → same packages → same build environment
- Different machines produce identical dev shells
- CI uses the same environment as local development
- No "works on my machine" syndrome

### When to Update Lock Files

- **Deliberately**: When a new package version is needed
- **Security patches**: When a CVE is fixed in a newer version
- **Breaking changes**: When updating requires code changes
- **Not automatically**: Lock files should not auto-update in CI
