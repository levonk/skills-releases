# Hybrid Vault Storage

## Failure Mode

Secrets scattered across service configurations create duplication, inconsistent
rotation, and difficulty auditing. Using only point-of-use distribution makes
rotation require updating multiple locations.

## Practice

Adopt a **hybrid approach** with centralized per-client vault storage for shared
secrets and in-service storage for service-specific transient secrets.

### Per-Client Central Vault

All shared secrets stored in client-specific vault files:

```
levonk/active/02-config/ansible/inventories/group_vars/infrahub-levonk-all.vault.yml
```

Contains:
- Database passwords (PostgreSQL, Redis)
- API keys (Cloudflare, external services)
- Authentication credentials (admin passwords, service accounts)
- Encryption keys (JWT secrets, session secrets, storage encryption)
- VPN credentials (NordVPN, Tailscale auth keys)
- Service integration secrets (AI services, monitoring)

### In-Service Transient Secrets

Service-specific transient secrets stored within service configurations:
- JWT tokens (generated by Authelia, stored in Redis)
- Session keys (generated by web apps, stored in session storage)
- OAuth tokens (generated during flows, stored in service databases)
- Temporary encryption keys (generated for specific operations)

### Why Hybrid

- **Central vault**: Single source of truth, easy rotation, auditability
- **In-service transient**: Service isolation, lifecycle alignment, no vault bloat
- **Ansible native**: Leverages existing Ansible vault infrastructure

### Alternatives Rejected

- **Point-of-use distribution**: Secret duplication, hard rotation — rejected
- **External secret management** (HashiCorp Vault, AWS Secrets Manager):
  Overkill for current scale — rejected
- **Environment variables only**: No encryption, no audit — rejected
