### Git Repository Management Skill
- **Purpose**: Repository initialization, change organization, and commit management with secret scanning and rollback-safe ordering
- **Usage**: Delegate repo init and adoption-changeset commit to this skill — do NOT call `git init` / `git add` / `git commit` directly
- **Benefit**: Secret scanning before commit, vertical grouping of changes, mandatory commit body validation, pre/post auto-tags for rollback safety, automatic divergence handling on push
- **Reference**: [Git Repository Management Skill](../git-repository-management/SKILL.md)

#### Repository Initialization

When the target directory is not yet a git repository, `git-collect.sh` emits a `NOT_A_GIT_REPO` signal + exit 2. The AI agent then decides whether to run `git-repo-init.bash` in full CREATE mode (initializes structure + commits) or `--no-init-structure` mode (init-only), based on the directory's contents and the user's intent.

```bash
# 1. Detect whether init is needed
./<git-repository-management>/scripts/git-collect.sh /path/to/target
# → emits: NOT_A_GIT_REPO\n<exit 2> if not a repo

# 2. If the signal fired, initialize (AI decides scope)
bash ./<git-repository-management>/scripts/git-repo-init.bash /path/to/target                       # full CREATE
# or
bash ./<git-repository-management>/scripts/git-repo-init.bash --no-init-structure /path/to/target  # init-only

# 3. Re-collect after init
./<git-repository-management>/scripts/git-collect.sh /path/to/target
```

#### Adoption Changeset Commit

After all adoption changes are staged, commit them as one logical batch with rollback safety:

```bash
# Collect all data (changes + quality checks) for AI analysis
./<git-repository-management>/scripts/git-collect.sh --json .

# AI analyzes and groups changes into commits (typically one "project-adoption" commit)

# Execute the batch commit with auto pre/post tags
./<git-repository-management>/scripts/git-commit-batch.sh --slug project-adoption .

# Push commits + tags (script handles divergence — never manually rebase)
./<git-repository-management>/scripts/git-push.sh .   # optional, if remote configured
```

#### Why Direct `git init` / `git commit` Is Forbidden

Calling `git init`, `git add`, or `git commit` directly bypasses:
- **Secret scanning** — `git-collect.sh` runs secret-redaction checks before any commit lands
- **Vertical grouping** — `git-commit-batch.sh` enforces one logical change per commit with mandatory body validation
- **Rollback safety** — pre/post auto-tags (`pre-project-adoption-<sha>`, `post-project-adoption-<sha>`) enable `git-rollback.sh --to <tag>` if the adoption introduces regressions
- **No-AI-signatures rule** — `git-commit-batch.sh` rejects commit messages containing "Generated by", "Co-Authored-By", or any tool attribution boilerplate

### Ignorefile Manager Skill
- **Purpose**: Generate, reconcile, and audit all ignore files (`.gitignore`, `.dockerignore`, `.codeiumignore`, `.cursorignore`, `.aiexclude`, `.npmignore`, VS Code exclude settings, ripgrep config) from modular concern sources
- **Usage**: Delegate all ignore file generation to this skill — do NOT hand-write `.gitignore` or any other ignore file
- **Benefit**: Single source of truth for ignore patterns across git, docker, jj (`.jj/`), AI indexing tools, npm packaging, IDE, and search; patterns never diverge across outputs; reconciles orphan patterns from manual additions before regenerating
- **Reference**: [Ignorefile Manager Skill](../../../general/ignorefile-manager/SKILL.md)

#### Adoption Workflow

```bash
# 1. Reconcile — harvest orphan patterns from any existing deployed ignore files
uv run --script <ignorefile-manager>/scripts/generate_ignores.py reconcile --target . --auto-assign

# 2. Audit — check which outputs are missing or stale
uv run --script <ignorefile-manager>/scripts/generate_ignores.py audit --target .

# 3. Generate — dry-run first, then apply
uv run --script <ignorefile-manager>/scripts/generate_ignores.py generate --target . --dry-run
uv run --script <ignorefile-manager>/scripts/generate_ignores.py generate --target .

# 4. (Optional) Update VS Code workspace files and ripgrep config
uv run --script <ignorefile-manager>/scripts/generate_ignores.py workspace --target .
uv run --script <ignorefile-manager>/scripts/generate_ignores.py ripgrep
```

#### Coverage

ignorefile-manager generates the following outputs from small concern files in `assets/concerns/` (secrets, build-artifacts, os-files, editor-files, dependencies, ai-generated, dev-local, binaries, vcs-meta, logs, lockfiles) composed via `assets/outputs.yaml`:

| Output | Covers |
|--------|--------|
| `.gitignore` | git — secrets, build artifacts, OS files, editor files, dependencies, vcs-meta (incl. `.jj/`), dev-local, logs, binaries |
| `.dockerignore` | docker — same concerns minus negation patterns (docker has limited negation support) + `.docker/`, `Dockerfile`, `docker-compose*.yml` |
| `.codeiumignore` | Codeium AI indexing — secrets, build artifacts, dependencies, vcs-meta, dev-local, logs, binaries, ai-generated |
| `.cursorignore` | Cursor AI indexing — same as `.codeiumignore` |
| `.aiexclude` | Generic AI indexing — same as `.codeiumignore` |
| `.npmignore` | npm packaging — everything not needed at runtime + `src/`, `test/`, `docs/`, etc. |
| VS Code `files.exclude` / `search.exclude` / `files.watcherExclude` | IDE visibility, search scope, watcher scope (JSON glob syntax) |
| ripgrep `ripgrepignore` + `config` | Search scope — secrets, build artifacts, dependencies, vcs-meta, dev-local, logs, binaries, lockfiles |

#### Why Hand-Writing `.gitignore` Is Forbidden

Hand-writing `.gitignore` (or any other ignore file) duplicates ignorefile-manager's concern sources and diverges over time:
- Adding `result/` to `.gitignore` but forgetting `.codeiumignore` → AI indexes build exhaust
- Adding `.codegraph/` to VS Code's `files.exclude` but not `.gitignore` → git tracks IDE exhaust
- Patterns get duplicated across files; sections aren't alphabetized; comments explaining *why* a pattern exists are missing

ignorefile-manager's reconcile step catches manual additions that would otherwise be silently overwritten on the next generation.

### Readme Upsert Skill
- **Purpose**: Create or update the project's `README.md` — the human-facing entry point visible on GitHub and in the repo root. Handles both greenfield (create from template) and brownfield (preserve accurate sections, update stale ones) cases.
- **Usage**: Delegate all README.md creation/update to this skill — do NOT hand-write README content via heredocs in `adopt-project.sh` or `configure-*.sh`
- **Benefit**: Single source of truth for README structure, required sections, and the README↔AGENTS.md consistency check. Hand-written READMEs duplicate the template and bypass the consistency check; the two copies diverge over time. readme-upsert also runs `verify_consistency.py` to check project name match, no content duplication, and no wrong sections in either file.
- **Reference**: [Readme Upsert Skill](../../../ai/readme-upsert/SKILL.md)

#### Adoption Workflow

```bash
# 1. Ensure AGENTS.md exists FIRST (project-adopter step 10 / agent-file-upsert)
#    so readme-upsert can link to it and verify_consistency.py can check agreement.

# 2. Invoke readme-upsert — it runs its 5-phase workflow:
#    Phase 1: Repository Analysis (read existing README, AGENTS.md, docs/, analyze structure)
#    Phase 2: Generate README.md from references/README-project-root-template.md.tmpl
#    Phase 3: Upsert (create from scratch OR update stale sections, preserve accurate ones)
#    Phase 4: Cross-Reference Check (links to AGENTS.md, internal-docs/oos/, internal-docs/adr/)
#    Phase 5: Consistency Verification (scripts/verify_consistency.py {REPO_ROOT})
# See <readme-upsert>/SKILL.md for the full workflow.
```

#### Required Sections (owned by readme-upsert)

| Section | Purpose |
|---------|---------|
| Project name + overview | What it does, who it's for (1-2 paragraphs) |
| Quick Start | Copy-paste ready setup commands (clone, bootstrap, run) |
| Build/Test Commands | Day-to-day developer commands |
| Project Structure | Directory layout with one-line descriptions |
| AI Agent Documentation | Link to `AGENTS.md` |

**Optional sections** (included if relevant): Development Workflow, Testing, Package Management, Troubleshooting, Contributing, License.

#### Why Hand-Writing README.md Is Forbidden

Hand-writing README content via heredocs in `adopt-project.sh` or `configure-*.sh` duplicates readme-upsert's template and bypasses its consistency checks:
- The inline README and the readme-upsert template diverge over time (different required sections, different Quick Start commands, different Project Structure)
- The README↔AGENTS.md consistency check (`verify_consistency.py`) is skipped — project name can mismatch, content can be duplicated, wrong sections can leak into either file
- Brownfield projects get a from-scratch README instead of having accurate sections preserved and stale ones updated
- The semantic consistency check (README vs code vs AGENTS.md vs docs/) is skipped

readme-upsert's Phase 3 upsert logic preserves accurate sections of an existing README and only updates stale ones, which is critical for adopting existing projects without destroying user-facing documentation.

### AI Development Loop Integration

The **ai-development-loop** skill provides the systematic workflow:

1. **Foundation Check** - Ensure clean starting state
2. **Ticket Selection** - Grab next actionable ticket
3. **Start Work** - Mark ticket as in_progress
4. **High Quality** - Ensure upstream has adequate testing (mandatory per ADR 20260131001)
5. **Strategy** - Determine implementation approach (reuse, dependency, new package, external service)
6. **Implementation** - Do the actual work

8. **Completion** - Mark ticket ready/closed
9. **Commit & Loop** - Commit changes and repeat

### Project Detection (Using Project Detection Skill)

Use the dedicated project-detection skill for comprehensive analysis:

```bash
# Source the detection functions
DETECTION_SKILL_PATH="$(dirname "${BASH_SOURCE[0]}")/../project-detection"
source "$DETECTION_SKILL_PATH/scripts/detect-build-systems.sh"
source "$DETECTION_SKILL_PATH/scripts/detect-ci-cd-systems.sh"
source "$DETECTION_SKILL_PATH/scripts/detect-workspace-configs.sh"

# Detect all systems
build_systems=$(detect_systems "$PROJECT_PATH" "false")
ci_cd_systems=$(detect_ci_cd_systems "$PROJECT_PATH" "false")
workspace_configs=$(analyze_workspace_configs "$PROJECT_PATH" "$PROJECT_NAME" "false")

# Output results
echo "Detected build systems: $build_systems"
echo "Detected CI/CD: $ci_cd_systems"
echo "Workspace configs: $workspace_configs"
```

**Detection capabilities include:**
- **Build Systems**: npm, pnpm, cargo, poetry, go modules, etc.
- **CI/CD Platforms**: GitHub Actions, GitLab CI, CircleCI, etc.
- **Workspace Tools**: pnpm workspaces, npm workspaces, Nx (preferred per ADR 20260419001), Turbo (legacy), etc.
- **Development Tools**: ESLint, Prettier, Black, Ruff, Clippy, etc.

#### Loop Prevention and Safety
The project-adopter includes comprehensive loop prevention:

**Automatic Detection:**
- Detects if project-adopter is already running
- Prevents nested calls from the same process tree
- Checks for surgical-config execution conflicts
- Analyzes parent process and environment variables

**Control Options:**
```bash
# Disable loop prevention (use with caution)
./scripts/adopt-project.sh --no-loop-prevention

# Force adoption even if loops detected
./scripts/adopt-project.sh --force-adoption

# Verbose output for debugging
./scripts/adopt-project.sh --verbose
```

**Safety Features:**
- **Lock files**: `/tmp/.project-adopter.lock` and `/tmp/.surgical-config.lock`
- **PID tracking**: Monitors running processes to detect conflicts
- **Process tree analysis**: Prevents nested calls from same process tree
- **Automatic cleanup**: Removes stale lock files on exit
- **Non-breaking**: Loop prevention doesn't break core functionality
