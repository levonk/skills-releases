## XDG Paths

All CLI scripts and tools must use [XDG Base Directory Specification](https://specifications.freedesktop.org/basedir-spec/)
paths for file storage. Never write to `~/.<tool>rc`, `~/.<tool>/`, or `/tmp`
(except for truly ephemeral files).

### Path Table

| Data type | Path | Env var | Created by |
|-----------|------|---------|------------|
| Transient cache | `${XDG_CACHE_HOME:-$HOME/.cache}/<tool>/` | `XDG_CACHE_HOME` | Script on first run |
| Persistent state | `${XDG_DATA_HOME:-$HOME/.local/share}/<tool>/` | `XDG_DATA_HOME` | Script on first run |
| Configuration | `${XDG_CONFIG_HOME:-$HOME/.config}/<tool>/` | `XDG_CONFIG_HOME` | User or `--install` |
| Runtime (sockets, locks) | `${XDG_RUNTIME_DIR:-/run/user/$UID}/<tool>/` | `XDG_RUNTIME_DIR` | Script on first run |

### What goes where

| Data | Location | Can be deleted? |
|------|----------|-----------------|
| API response cache | Cache | Yes — regenerated on next call |
| Downloaded temp files | Cache | Yes — re-downloaded |
| Search index | Cache | Yes — rebuilt |
| Command history | Data | No — user loses history |
| SQLite database | Data | No — user loses data |
| User preferences | Config | No — user loses settings |
| Default config template | Config | Generated by `--install` |
| Unix socket | Runtime | Yes — recreated on restart |
| PID file | Runtime | Yes — recreated on restart |

### Python resolution

```python
import os
from pathlib import Path

def cache_dir(tool: str) -> Path:
    base = os.environ.get("XDG_CACHE_HOME", str(Path.home() / ".cache"))
    return Path(base) / tool

def data_dir(tool: str) -> Path:
    base = os.environ.get("XDG_DATA_HOME", str(Path.home() / ".local" / "share"))
    return Path(base) / tool

def config_dir(tool: str) -> Path:
    base = os.environ.get("XDG_CONFIG_HOME", str(Path.home() / ".config"))
    return Path(base) / tool

# Always mkdir -p before writing
data_dir("mytool").mkdir(parents=True, exist_ok=True)
```

### Bash resolution

```bash
CACHE_DIR="${XDG_CACHE_HOME:-$HOME/.cache}/<tool>"
DATA_DIR="${XDG_DATA_HOME:-$HOME/.local/share}/<tool>"
CONFIG_DIR="${XDG_CONFIG_HOME:-$HOME/.config}/<tool>"
RUNTIME_DIR="${XDG_RUNTIME_DIR:-/run/user/$(id -u)}/<tool>"

mkdir -p "$CACHE_DIR" "$DATA_DIR"
```

### Rust resolution

```rust
use std::path::PathBuf;

fn cache_dir(tool: &str) -> PathBuf {
    let base = std::env::var("XDG_CACHE_HOME")
        .unwrap_or_else(|_| {
            dirs::cache_dir()
                .map(|p| p.to_string_lossy().into_owned())
                .unwrap_or_else(|| "~/.cache".to_string())
        });
    PathBuf::from(base).join(tool)
}
```

Use the [`dirs`](https://crates.io/crates/dirs) crate for platform-native paths.

## Output Discipline

### stdout vs stderr

| Channel | Content | Read by agent |
|---------|---------|---------------|
| stdout | Data, structured errors, suggestions | Yes |
| stderr | Debug logs, progress, diagnostics | No |

**Never mix progress messages into stdout.** An agent that reads "Fetching
data..." will try to interpret it as data.

### Color control (full CLI tools)

Support `--color=auto|always|never` (default: auto). Honor `NO_COLOR` env var
(takes precedence over all other settings).

Precedence: `NO_COLOR` env > `--color` flag > config file > auto-detection.

### JSON output (full CLI tools)

Provide `--json` flag for machine-readable output. When set:
- Output pure JSON to stdout (no color, no progress)
- Errors as JSON objects: `{"error": "msg", "help": "suggestion"}`
- Exit codes unchanged

## Configuration Precedence

Full CLI tools follow this precedence (highest to lowest):

1. CLI args
2. Environment variables (`MYTOOL_*` prefix)
3. Local project config (`./.mytool.toml`)
4. User config (`${XDG_CONFIG_HOME:-$HOME/.config}/mytool/config.toml`)
5. System/enterprise config (`/etc/mytool/config.toml`)
6. Hardcoded defaults

Prefer TOML for human-edited config; YAML for complex structures; JSON for
machine-to-machine.

### Config file initialization

On first run, if no config file exists in the expected location, create a
default config file with all settings commented out, including default values
and explanations for each option.

## Security

### Secrets

- Never log secrets, API keys, tokens, or passwords
- Mask secrets in error messages: `api_key: ****`
- Read secrets from env vars or system keyring — never from config files in
  plaintext
- Never echo the full secret in `--verbose` mode

### Input validation

- Validate all input at trust boundaries (CLI args, stdin, env vars, config
  files)
- Reject unknown flags and arguments (AXI principle 12)
- Validate file paths against directory traversal (`../`)
- Sanitize any input used in shell commands (use `subprocess` with arg lists,
  never `shell=True` with user input)

### File permissions

- Config files: `0600` (owner read/write only) if they contain secrets
- Data files: `0644` (owner read/write, others read)
- Cache files: `0644`
- Runtime sockets: `0600`

### Dependency safety

- Prefer stdlib over third-party deps
- When third-party deps are needed, pin versions in PEP 723 header (Python) or
  lock file (Rust/Go/Node)
- Never `pip install` at runtime — use `uv run --script` with PEP 723
- Check packages with `deptrust` before adding if available

## Full CLI Tool Scaffolding

Full CLI tools are scaffolded from `levonk-base-boilerplate` templates using
the copier wrapper.

### Available templates

| Language | Path | Framework |
|----------|------|-----------|
| Python | `apps/cli/python/core/` | argparse (stdlib) or click (optional) |
| Rust | `apps/cli/rust/core/` | clap |
| Go | `apps/cli/go/core/` | cobra |
| TypeScript | `apps/cli/typescript/core/` | commander |
| Bash | `apps/cli/bash/core/` | stdlib |
| Others | `apps/cli/<lang>/core/` | idiomatic |

### Scaffolding steps

1. **Prepare copier answers:**
   ```bash
   cat > copier-answers.yml <<EOF
   project_name: my-cli-tool
   package_slug: my-cli-tool
   description: "A CLI tool for doing X"
   # Add language-specific variables
   EOF
   ```

2. **Generate the project:**
   ```bash
   devbox run -- rtk ./boilerplate/copier-wrapper.sh copy \
     ./boilerplate/apps/cli/<language>/core \
     ./path/to/new-cli-tool \
     --data @copier-answers.yml
   ```

3. **Post-scaffold — the boilerplate covers:**
   - `--help`/`-h`, `--version`/`-v` (framework built-in)
   - Config file initialization
   - Shell completion (framework-generated)
   - Dockerfile and container hardening
   - devbox.json and flake.nix
   - Justfile targets (build, test, lint, format)
   - Pre-commit hooks
   - Project.json (Nx)

4. **Post-scaffold — the skill adds:**
   - AXI agent mode (structured output, TOON, minimal schemas)
   - XDG path resolution
   - Session hook infrastructure (`--install-agent-hooks`)
   - Content-first no-args behavior
   - Contextual disclosure (`help[]` suggestions)
   - `--json` and `--toon` output modes
   - `--fields` flag for schema selection
   - `--full` flag for content truncation escape hatch
   - `--human`/`--interactive` flags for human mode
   - Mode auto-detection (TTY → human, no-TTY → agent)

5. **Validate:**
   ```bash
   cd ./path/to/new-cli-tool
   just build
   just test
   just lint
   ```

## Testing

### Embedded scripts

- Test with direct invocation: `uv run --script scripts/foo.py --arg value`
- Test quiet mode (default): verify minimal output
- Test `--verbose`: verify full detail
- Test `--dry-run`: verify no side effects
- Test error paths: missing required args, invalid input, dependency missing
- Test exit codes: 0 for success/no-op, 1 for error, 2 for usage error
- Test with no TTY (pipe output): verify no interactive prompts

### Full CLI tools

- All embedded script tests, plus:
- Test `--help` output matches command structure
- Test `--version` output
- Test `--json` output is valid JSON
- Test `--toon` output is valid TOON
- Test no-args shows content (not just help)
- Test `--fields` flag selects correct fields
- Test `--full` disables truncation
- Test `--color=auto|always|never` and `NO_COLOR`
- Test config file initialization on first run
- Test config precedence (args > env > project > user > system > defaults)
- Test shell completion generation (`--install`)
